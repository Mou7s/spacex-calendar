Recommended Next Work

下一步做什么

这个项目的核心已经清楚：从 SpaceX 官网数据源合并任务和发射窗口，输出稳定 ICS， 再用轻量静态页展示。接下来最值得做的是降低维护成本、提高订阅可靠性，并补齐可验证的用户体验。

1 优先保护 ICS 稳定性，这是项目最核心的用户价值。
2 把前端翻译和渲染拆小，后续功能会更容易加。
3 用 E2E 覆盖订阅、日历浏览、错误态这些真实路径。
4 谨慎扩功能，避免把静态落地页做重。

Phase 1

近期最推荐

这些改动风险低、收益直接，适合优先排进下一轮。

P0 / Now

为 ICS 输出补更严格的回归测试

重点覆盖 UID 不变化、Content-Type、DTSTART/DTEND、转义、折行、无时间任务跳过等行为。 这是订阅型产品的地基，后续改 UI 或数据源都不能影响日历客户端同步。

价值：防破坏订阅 验收：npm test 覆盖关键 ICS 字段 范围：test/spacex.test.js
P0 / Now

拆出前端翻译数据

当前中英文文案集中在 app.js，文件会随着功能增长越来越难读。建议拆成独立 translations 模块，仍然保持浏览器原生 ES module，不引入框架。

价值：降低维护成本 验收：中英文页面无回退 key 范围：public/app.js
P1 / Now

给上游数据异常做可观察性

保留现有 SpaceX 数据源，但在 API 错误响应里区分 tiles 源、timings 源、JSON 结构变化、 KV 读写失败。这样线上出问题时能快速判断是上游改版还是缓存/Worker 问题。

价值：更快定位故障 验收：错误测试覆盖来源标记 范围：src/lib/spacex.js

Phase 2

中期增强

这些会明显提升体验，但建议在核心稳定后再做。

P1 / Next

补 Playwright E2E 测试

覆盖首页加载、语言切换后的关键文案、订阅链接生成、月历翻页、任务卡片渲染、API 失败态。 这比单元测试更接近真实用户路径。

价值：防 UI 回归 验收：本地和 CI 可跑 E2E 范围：test/e2e
P1 / Next

月历日期点击联动任务列表

当前已有月历和任务列表，下一步可以让点击某个发射日期时滚动并高亮对应任务。 这是小功能，但会让页面从“展示”更接近“可浏览工具”。

价值：提升浏览效率 验收：键盘和鼠标都可触发 范围：public/app.js
P2 / Next

增加手动时区选择

默认使用浏览器时区很好，但跨地区关注发射的人可能想看 UTC、美国东部时间或发射场当地时间。 建议只影响页面展示，不改变 ICS 里的 UTC 时间。

价值：适合跨时区用户 验收：刷新后保留选择 范围：public/app.js

Phase 3

谨慎评估

这些功能有吸引力，但会带来权限、数据或复杂度成本。

P2 / Later

单任务分享链接

可以为任务卡片生成带 hash 的链接，并在打开后自动定位到任务。比社交分享 API 更轻， 也更符合静态页面路线。

价值：传播单个任务 验收：URL 可直达任务卡片 范围：public/app.js
P2 / Later

发射历史归档

如果 SpaceX 仍提供稳定历史数据源，可以增加“已完成”标签页。不要为了归档引入数据库， 除非明确要做长期历史产品。

价值：补全浏览场景 验收：数据源稳定且测试覆盖 范围：数据源待确认
P3 / Watch

浏览器发射提醒通知

Web Notification API 听起来很适合，但权限请求、离线调度、Service Worker 唤醒能力都有限。 只有在明确用户需要网页端提醒时再做，且不要影响 ICS 订阅主路径。

价值：可选提醒 验收：用户主动开启 范围：public/sw.js

执行顺序建议

  • 先补 ICS 回归测试，再调整任何 Worker 或数据规范化逻辑。
  • 前端先拆翻译，再做月历联动和时区选择，避免在大 app.js 里继续堆功能。
  • 每次触碰 Worker 逻辑后跑 node --check src/index.js、node --check src/lib/spacex.js 和 npm test。
  • 每次触碰前端逻辑后跑 node --check public/app.js 和 npm test。
